## 1. 개요

### Fastpath란?

**Fastpath**는 PostgreSQL B-tree 인덱스에서 **단조 증가(montonically increasing) 키 삽입 시 트리 탐색을 생략하는 최적화 기법**입니다.

```
일반 삽입: O(log N) - 루트부터 리프까지 탐색
Fastpath:   O(1)     - 캐시된 페이지 바로 접근
```

### 관련 상수

```c
/* Minimum tree height for application of fastpath optimization */
#define BTREE_FASTPATH_MIN_LEVEL    2
```

## 2. 문제 정의

### 일반적인 B-tree 삽입 과정

```
        [루트: 50]
       /           \
   [25]           [75]
   /   \         /    \
[10]  [40]    [60]   [90]
                    ↑
              새 데이터 95 삽입
```

**일반적인 과정 (O(log N)):**
1. 루트에서 이진 탐색 → 75로 이동
2. 내부 노드에서 이진 탐색 → 90으로 이동
3. 리프 노드 도달 → 삽입 위치 찾기
4. 데이터 삽입

### 현실적 관찰

많은 워크로드에서 **모든 삽입이 인덱스의 끝에서 발생**합니다:

| 패턴 | 예시 | 특성 |
|------|------|------|
| AUTO_INCREMENT | `id SERIAL PRIMARY KEY` | 항상 증가 |
| TIMESTAMP | `created_at TIMESTAMP DEFAULT NOW()` | 시간 순서 |
| SEQUENCE | `nextval('order_seq')` | 순차 증가 |

```
시간 흐름에 따른 삽입:
┌─────────────────────────────────────────────────┐
│  1 → 2 → 3 → 4 → 5 → ... → 9999 → 10000 → ...  │
│                                                 │
│  모든 삽입이 "가장 오른쪽"에서 발생!            │
└─────────────────────────────────────────────────┘
```

**핵심 통찰**: 매번 루트부터 탐색하는 것은 낭비!

## 3. Fastpath 동작 원리

### 핵심 아이디어

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   이전 삽입에서 "가장 오른쪽 리프 페이지"를 캐싱해두고      │
│                                                             │
│   다음 삽입 시:                                             │
│   1. 캐시된 페이지가 여전히 "가장 오른쪽"인지 확인         │
│   2. 새 키가 이 페이지에 들어갈 수 있는지 확인             │
│   3. 조건 만족 시 → 바로 삽입 (트리 탐색 생략!)            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 시각화

```
=== 일반 삽입 (Fastpath 미사용) ===

         [루트]
        /      \
     [중간]   [중간]
      /  \
   [리프] [리프]
            ↑
      루트부터 탐색 필요 (3단계)


=== Fastpath 삽입 ===

         [루트]
        /      \
     [중간]   [중간]
      /  \
   [리프] [리프]  ← 캐시된 페이지!
            ↑
      바로 접근 (1단계)
```

### 캐싱 메커니즘

```c
// 삽입 완료 후 캐싱 설정
if (P_RIGHTMOST(lpageop) && P_ISLEAF(lpageop) && !P_ISROOT(lpageop))
    cachedBlock = BufferGetBlockNumber(buf);

// 트리 높이 조건 확인 후 저장
if (BlockNumberIsValid(cachedBlock) &&
    _bt_getrootheight(rel) >= BTREE_FASTPATH_MIN_LEVEL)
    RelationSetTargetBlock(rel, cachedBlock);
```

## 4. 상세 조건 분석

### Fastpath 진입 5가지 조건

```c
if (P_ISLEAF(lpageop) && P_RIGHTMOST(lpageop) &&
    !P_IGNORE(lpageop) &&
    (PageGetFreeSpace(page) > itemsz) &&
    PageGetMaxOffsetNumber(page) >= P_FIRSTDATAKEY(lpageop) &&
    _bt_compare(rel, indnkeyatts, itup_scankey, page,
                P_FIRSTDATAKEY(lpageop)) > 0)
{
    fastpath = true;
}
```

### 조건 분석표

```
┌────┬─────────────────────────┬─────────────────────────────────┐
│ #  │ 조건                     │ 이유                            │
├────┼─────────────────────────┼─────────────────────────────────┤
│ 1  │ P_ISLEAF(lpageop)       │ 리프 페이지여야 함              │
│    │                         │ (내부 노드는 탐색용)            │
├────┼─────────────────────────┼─────────────────────────────────┤
│ 2  │ P_RIGHTMOST(lpageop)    │ 가장 오른쪽 페이지여야 함       │
│    │                         │ (단조 증가 키는 항상 끝으로)    │
├────┼─────────────────────────┼─────────────────────────────────┤
│ 3  │ !P_IGNORE(lpageop)      │ 삭제/삭제표시된 페이지 제외     │
│    │                         │ (일관성 보장)                   │
├────┼─────────────────────────┼─────────────────────────────────┤
│ 4  │ PageGetFreeSpace > sz   │ 새 아이템이 들어갈 공간 있어야  │
│    │                         │ (분할 피하기)                   │
├────┼─────────────────────────┼─────────────────────────────────┤
│ 5  │ _bt_compare(...) > 0    │ 삽입 키 > 페이지 첫 번째 키     │
│    │                         │ (단조 증가 확인)                │
└────┴─────────────────────────┴─────────────────────────────────┘
```

### 조건 5 상세 설명

```
페이지 구조:
┌─────────────────────────────────────────────────┐
│ [High Key=100] [50] [60] [70] [80] [90] [95]    │
│  ↑                                              │
│  첫 번째 키 (P_FIRSTDATAKEY)                    │
└─────────────────────────────────────────────────┘

삽입 키 = 96
비교: 96 > 50? → YES → Fastpath 진입 가능!

삽입 키 = 45
비교: 45 > 50? → NO → Fastpath 불가 (중간 삽입)
```

## 5. 락 경쟁 완화 전략

### 문제: 락 핫스팟

Fastpath는 모든 삽입이 **동일한 페이지(가장 오른쪽 리프)**로 몰리게 합니다:

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   프로세스 A ──┐                                           │
│               │                                             │
│   프로세스 B ──┼──→ [가장 오른쪽 리프] ←── 락 경쟁 발생!  │
│               │                                             │
│   프로세스 C ──┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 해결책: 조건부 락 (Conditional Lock)

```c
if (ConditionalLockBuffer(buf))
{
    // 락 획득 성공 → fastpath 진행
}
else
{
    ReleaseBuffer(buf);
    RelationSetTargetBlock(rel, InvalidBlockNumber);
    // 락 획득 실패 → 일반 경로로 fallback
}
```

### 동작 논리

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ConditionalLockBuffer() 시도                               │
│                                                             │
│  ├── 성공: Fastpath 진행                                    │
│  │                                                          │
│  └── 실패: 다른 프로세스가 이미 락 보유                     │
│      │                                                      │
│      ├── 캐시 무효화                                        │
│      │                                                      │
│      └── 일반 경로(_bt_search)로 이동                       │
│          (다른 페이지 사용 가능성)                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 왜 이것이 효과적인가?

```
락 대기 vs Fallback 비교:

┌─────────────────────────────────────────────────────────────┐
│ 락 대기 (기존 방식)                                         │
│ ├── 모든 프로세스가 동일 페이지 대기                        │
│ ├── 순차 처리 (직렬화)                                      │
│ └── 전체 처리량 저하                                        │
├─────────────────────────────────────────────────────────────┤
│ Conditional + Fallback (개선)                               │
│ ├── 락 없으면 즉시 다른 경로 시도                           │
│ ├── 병렬 처리 가능                                          │
│ └── 전체 처리량 향상                                        │
└─────────────────────────────────────────────────────────────┘
```

## 6. 성능 효과

### 복잡도 비교

| 시나리오 | 일반 경로 | Fastpath |
|----------|-----------|----------|
| 트리 탐색 | O(log N) | O(1) |
| 페이지 락 | 여러 페이지 | 1개 페이지 |
| 디스크 I/O | 여러 페이지 읽기 | 1개 페이지 |

### 트리 높이 제한

```c
// 트리 높이가 2 미만이면 Fastpath 비활성화
if (_bt_getrootheight(rel) >= BTREE_FASTPATH_MIN_LEVEL)
    RelationSetTargetBlock(rel, cachedBlock);
```

**이유**: 작은 인덱스는 트리 탐색 비용이 낮아 Fastpath 오버헤드가 더 클 수 있음

```
트리 높이별 비교:

높이 1 (루트 = 리프):
├── 일반 탐색: 1단계
├── Fastpath: 캐싱 오버헤드
└── 결론: Fastpath 비활성화

높이 2+:
├── 일반 탐색: 2+단계
├── Fastpath: 1단계
└── 결론: Fastpath 활성화
```

```
│     └── P_IGNORE 플래그 설정                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 8. 실제 사용 사례

### 적합한 시나리오

```sql
-- 시나리오 1: AUTO_INCREMENT
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,        -- 단조 증가
    customer_id INT,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 시나리오 2: 타임스탬프 기반
CREATE TABLE logs (
    id SERIAL,
    log_time TIMESTAMP DEFAULT NOW(),  -- 시간 순서
    message TEXT
);
CREATE INDEX idx_log_time ON logs(log_time);

-- 시나리오 3: 시퀀스 사용
CREATE SEQUENCE order_seq;
CREATE TABLE orders (
    order_id BIGINT DEFAULT nextval('order_seq'),
    ...
);
```

### 부적합한 시나리오

```sql
-- 시나리오 1: 랜덤 키
CREATE TABLE users (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,  -- 랜덤
    name TEXT
);

-- 시나리오 2: 역순 삽입
CREATE TABLE events (
    id SERIAL,
    event_time TIMESTAMP,
    -- event_time DESC 인덱스는 fastpath 비효율
);
CREATE INDEX idx_event_time_desc ON events(event_time DESC);

-- 시나리오 3: 중간 삽입
-- 기존 키 사이에 새로운 키 삽입이 빈번한 경우
```

### 성능 벤치마크 (개념적)

```
단조 증가 키 삽입 (1,000,000 레코드):
```

## 요약

| 항목 | 설명 |
|------|------|
| **목적** | 단조 증가 키 삽입 시 트리 탐색 생략 |
| **원리** | 가장 오른쪽 리프 페이지 캐싱 |
| **조건** | 리프 + 오른쪽 + 공간있음 + 단조증가키 + 높이≥2 |
| **복잡도** | O(log N) → O(1) |
| **장점** | 삽입 성능 대폭 향상 (최대 70%+) |
| **단점** | 오른쪽 리프 페이지 락 경쟁 가능성 |
| **완화** | 조건부 락으로 경쟁 시 fallback |
