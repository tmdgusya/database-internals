# B-Tree 학습 가이드

> 이 문서는 B-Tree의 개념과 알고리즘을 설명합니다. 코드를 직접 작성하며 학습할 수 있도록 구조와 동작 원리만 설명합니다.

---

## 1. 정의 (Definition)

B-Tree는 **자가 균형 트리(Self-balancing Tree)** 로, 데이터베이스와 파일 시스템에서 널리 사용되는 자료구조입니다.

### 핵심 특징
- **다진 트리(M-ary Tree)**: 각 노드가 여러 개의 자식을 가질 수 있음
- **균형 유지**: 모든 리프 노드가 같은 깊이에 위치
- **디스크 친화적**: 노드 하나를 디스크 블록 하나에 매핑 가능 → I/O 횟수 최소화

---

## 2. B-Tree의 속성 (Properties)

차수(Order)가 **m**인 B-Tree의 속성:

### 노드 구조
```
각 노드는 다음을 포함:
- 최대 m개의 자식
- 최대 m-1개의 키
- 최소 ⌈m/2⌉개의 자식 (루트 제외)
- 최소 ⌈m/2⌉ - 1개의 키 (루트 제외)
```

### 루트 노드 예외
- 루트는 자식이 최소 2개 (리프가 아닌 경우)
- 루트는 키가 최소 1개

### 리프 노드
- 모든 리프 노드는 같은 깊이
- 리프 노드의 키 개수: ⌈m/2⌉ - 1 ~ m - 1

### 키 정렬
- 각 노드 내의 키는 오름차순 정렬
- 키 k[i]에 대해: 왼쪽 서브트리의 모든 키 < k[i] < 오른쪽 서브트리의 모든 키

### 시각적 예시 (m = 4, 즉 4차 B-Tree)
```
                    [30 | 60]
                   /    |    \
          [10 | 20]  [40 | 50]  [70 | 80 | 90]
          
- 루트: 키 2개 (30, 60), 자식 3개
- 각 내부 노드: 키 2-3개, 자식 3-4개
- 모든 리프가 같은 레벨
```

## 3. 탐색 (Search) 알고리즘

### 동작 원리

1. 루트 노드부터 시작
2. 현재 노드에서 탐색 키와 비교
3. 탐색 키가 노드 내에 있으면 → **발견**
4. 없으면 적절한 자식 노드로 이동:
   - `key < k[0]` → 첫 번째 자식
   - `k[i] < key < k[i+1]` → i+1 번째 자식
   - `key > k[last]` → 마지막 자식
5. 리프 노드까지 도달했는데 없으면 → **없음**

### 의사코드
```
function search(node, key):
    if node is null:
        return NOT_FOUND
    
    # 현재 노드에서 키 찾기
    i = 0
    while i < node.key_count and key > node.keys[i]:
        i = i + 1
    
    # 키를 찾은 경우
    if i < node.key_count and key == node.keys[i]:
        return (node, i)  # 노드와 인덱스 반환
    
    # 리프 노드면 찾기 실패
    if node.is_leaf():
        return NOT_FOUND
    
    # 자식 노드로 재귀
    return search(node.children[i], key)
```

### 시간 복잡도
- **O(log n)**: 트리 높이가 log n에 비례
- 높이 h = O(log_m n) (m은 차수, n은 키 개수)

## 4. 삽입 (Insertion) 알고리즘

### 핵심 원칙
- **항상 리프 노드에 삽입**
- 오버플로우 발생 시 **분할(Split)** 수행

### 동작 원리

1. **삽입 위치 찾기**: 탐색 알고리즘으로 리프 노드 찾기
2. **리프 노드에 삽입**: 키를 정렬된 위치에 삽입
3. **오버플로우 확인**: 키 개수가 m-1을 초과하면 분할

### 분할 (Split) 과정

노드가 꽉 찼을 때 (키가 m개):

```
분할 전 (m=4, 키 4개 = 오버플로우):
[10 | 20 | 30 | 40]

분할 후:
        [30]              ← 중간 키가 부모로 승진
       /    \
   [10 | 20]  [40]        ← 좌우로 분할
```

**분할 규칙**:
1. 중간 키(median)를 부모 노드로 승진
2. 왼쪽 키들은 새로운 왼쪽 노드
3. 오른쪽 키들은 새로운 오른쪽 노드
4. 부모 노드도 오버플로우면 재귀적으로 분할

### 의사코드
```
function insert(node, key):
    if node is null:
        return create_node(key)
    
    # 리프 노드면 삽입
    if node.is_leaf():
        insert_key_in_node(node, key)
        
        # 오버플로우 확인
        if node.key_count > m - 1:
            return split(node)
        return node
    
    # 리프가 아니면 적절한 자식으로 이동
    i = find_child_index(node, key)
    child_result = insert(node.children[i], key)
    
    # 자식이 분할되었으면 부모 중간 키 추가
    if child_result is split_result:
        insert_key_in_node(node, child_result.median_key)
        update_children(node, child_result)
        
        if node.key_count > m - 1:
            return split(node)
    
    return node
```

### 삽입 예시 (m = 3)

```
빈 트리에 10, 20, 5, 6, 12, 30, 7, 17 순서로 삽입

1. 10 삽입:
   [10]

2. 20 삽입:
   [10 | 20]

3. 5 삽입:
   [5 | 10 | 20]

4. 6 삽입 → 오버플로우! 분할:
         [10]
        /    \
      [5 | 6]  [20]

5. 12 삽입:
         [10]
        /    \
      [5 | 6]  [12 | 20]

6. 30 삽입:
         [10]
        /    \
      [5 | 6]  [12 | 20 | 30]

7. 7 삽입 → 오버플로우! 분할:
         [10 | 7]
        /    |    \
      [5 | 6] [7]  [12 | 20 | 30]
      
      (실제로는 7이 올라가고 재정렬)
      
         [7 | 10]
        /    |    \
      [5 | 6] [7]  [12 | 20 | 30]
      
      (다시 오버플로우 체크)

... 계속
```

## 5. 삭제 (Deletion) 알고리즘

### 핵심 원칙
- **언더플로우 방지**: 삭제 후 키 개수가 최소보다 적어지면 재조정
- 내부 노드의 키 삭제 시 **선임자(predecessor) 또는 후임자(successor)** 로 대체

### 동작 원리

#### Case 1: 리프 노드에서 삭제
```
1. 키 삭제
2. 키 개수가 최소(⌈m/2⌉ - 1) 이상이면 완료
3. 최소 미만이면 언더플로우 → 재조정 필요
```

#### Case 2: 내부 노드에서 삭제
```
1. 삭제할 키의 선임자(왼쪽 서브트리의 최댓값) 또는
   후임자(오른쪽 서브트리의 최솟값) 찾기
2. 선임자/후임자로 삭제할 키 대체
3. 선임자/후임자가 있던 리프 노드에서 삭제
4. 리프 노드에서 언더플로우 발생 가능 → 재조정
```

### 재조정 (Rebalancing) 전략

#### 전략 1: 빌려오기 (Borrow from Sibling)
형제 노드에 여유 키가 있으면 빌려옴

```
빌려오기 전:
        [20]
       /     \
    [10]    [30 | 40 | 50]   ← 오른쪽 형제에 여유 있음
    
[10] 삭제 후 언더플로우, 빌려오기:
        [30]
       /     \
    [20]    [40 | 50]
```

#### 전략 2: 병합 (Merge with Sibling)
형제도 여유가 없으면 병합

```
병합 전:
        [20 | 40]
       /    |    \
    [10]  [30]  [50]   ← 모두 최소 키만 가짐
    
[10] 삭제 후 언더플로우, 병합:
        [40]
       /     \
    [20 | 30]  [50]
```

### 의사코드
```
function delete(node, key):
    if node is null:
        return NOT_FOUND
    
    # 키 찾기
    i = find_key_index(node, key)
    
    if key found in node:
        if node.is_leaf():
            # Case 1: 리프에서 직접 삭제
            remove_key(node, i)
            return handle_underflow(node) if needed
        else:
            # Case 2: 내부 노드 - 후임자로 대체
            successor = find_successor(node.children[i+1])
            node.keys[i] = successor
            delete(node.children[i+1], successor)
    else:
        # 자식으로 이동
        child_idx = find_child_index(node, key)
        delete(node.children[child_idx], key)
    
    # 자식 삭제 후 언더플로우 처리
    return rebalance(node, child_idx) if needed
```

### 삭제 예시 (m = 3)

```
초기 상태:
         [20]
        /    \
    [10 | 15]  [30 | 40]

1. 15 삭제 (리프, 최소 유지):
         [20]
        /    \
      [10]    [30 | 40]

2. 10 삭제 (리프, 언더플로우!):
   - 형제 [30 | 40]에서 빌려오기
   
         [30]
        /    \
      [20]    [40]

3. 20 삭제 (내부 노드):
   - 후임자 30으로 대체
   - 리프에서 30 삭제
   
         [40]
        /
      [30]
      
   (실제로는 루트가 변경될 수 있음)
```

## 6. 리밸런싱 (Rebalancing) 상세

### 분할 (Split) - 삽입 시

**언제?** 노드의 키 개수가 m개가 되면

**어떻게?**
```
1. 중간 키(median) 선택: keys[⌊m/2⌋]
2. 새 노드 생성
3. 중간 키 기준으로 분할:
   - 왼쪽: keys[0..median-1]
   - 오른쪽: keys[median+1..m-1]
4. 중간 키를 부모로 승진
5. 부모도 오버플로우면 재귀 분할
```

### 병합 (Merge) - 삭제 시

**언제?** 삭제 후 키 개수가 ⌈m/2⌉ - 2개가 되면

**어떻게?**
```
1. 왼쪽 또는 오른쪽 형제 선택
2. 부모의 구분 키를 가져옴
3. 형제의 모든 키와 합침
4. 부모에서 구분 키 제거
5. 부모도 언더플로우면 재귀 병합
```

### 빌려오기 (Borrow) - 삭제 시

**언제?** 삭제 후 언더플로우, 형제에 여유 키가 있을 때

**어떻게?**
```
왼쪽 형제에서 빌려오기:
1. 부모의 구분 키를 현재 노드의 첫 번째로 이동
2. 왼쪽 형제의 마지막 키를 부모로 이동
3. 회전 효과

오른쪽 형제에서 빌려오기:
1. 부모의 구분 키를 현재 노드의 마지막으로 이동
2. 오른쪽 형제의 첫 번째 키를 부모로 이동
```

## 8. B+ Tree와의 차이점 (참고)

데이터베이스에서는 B+ Tree를 더 많이 사용합니다:

| 특징 | B-Tree | B+ Tree |
|------|--------|---------|
| 데이터 저장 | 모든 노드에 저장 | 리프 노드에만 저장 |
| 리프 연결 | 없음 | 연결 리스트로 연결 |
| 중복 키 | 없음 | 내부 노드에 중복 가능 |
| 범위 검색 | 비효율적 | 효율적 (리프 순회) |

## 9. 추천 학습 순서

1. **노드 구조 구현** → 단위 테스트
2. **탐색 구현** → 단위 테스트
3. **삽입 구현** (분할 없이) → 단위 테스트
4. **분할 구현** → 삽입과 통합 → 테스트
5. **삭제 구현** (재조정 없이) → 단위 테스트
6. **빌려오기 구현** → 삭제와 통합 → 테스트
7. **병합 구현** → 삭제와 통합 → 테스트
8. **종합 테스트** → 대량 데이터, 엣지 케이스
