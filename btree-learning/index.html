Why PostgreSQL Uses B-link-Tree PostgreSQL

🔓

Lock-Free Reads

Readers don't need to acquire locks on nodes during traversal, enabling true concurrent access even during structural modifications.

⚡

No Reader Starvation

With traditional B-Trees, heavy write workloads can starve readers. B-link-Tree ensures readers always make progress.

🔄

High Concurrency

Multiple readers and writers can operate simultaneously without blocking each other in most scenarios.

🛡️

Crash Safety

The right-link mechanism provides a consistent view of the tree structure even during incomplete operations.

How the Lock-Free Traversal Works

🔍 B-link-Tree Search Algorithm
function search(node, key): while node != null: // Search within current node i = find_position(node.keys, key) if i < node.key_count and node.keys[i] == key: return FOUND // Check if we need to follow right-link (split occurred) if key >= node.high_key: // Node was split, follow right-link node = node.right_link continue if node.is_leaf: return NOT_FOUND // Move to appropriate child node = node.children[i] return NOT_FOUND

The Split Process with Right-Links

When a node splits in a B-link-Tree:

  1. A new right sibling node is created with half the keys
  2. The right-link of the original node points to the new sibling
  3. The high key of the original node is updated to the separator value
  4. Readers looking for keys ≥ high key automatically follow the right-link
  5. The parent node is updated with the new separator (can be deferred)

💡 Key Insight

The high key acts as a "fence" - if a search encounters a key larger than the high key, it knows the node has been split and follows the right-link to find the data. This allows readers to always find their target without waiting for splits to complete.
