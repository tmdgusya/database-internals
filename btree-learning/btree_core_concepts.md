# B-Tree 핵심 개념: 수학적 정의와 분석

---

## 1. B-Tree의 정의와 핵심 성질

### 1.1 직관적 이해: 왜 B-Tree가 필요한가?

**문제 상황**: 디스크와 같은 외부 저장장치에서 데이터를 효율적으로 관리하려면?

- 디스크는 블록 단위로 데이터를 읽고 쓴다 (일반적으로 4KB)
- 이진 트리는 한 노드에 하나의 키만 저장 → 디스크 접근이 너무 잦음
- **해결책**: 한 노드에 여러 키를 저장하여 디스크 접근 횟수를 줄인다!

> B-Tree는 "팬아웃(fan-out)"이 큰 균형 트리로, 한 노드에 여러 키를 저장하여 디스크 I/O를 최소화한다.

---

### 1.2 구체적 예시: t=3인 B-Tree (2-3-4 Tree)

```
                    [30, 60]
                   /    |    \
            [10,20]  [40,50]  [70,80,90]
            /  |  \   /  |  \  /  |  |  \
           ...  ...  ... ... ... ... ... ...
```

**이 예시에서 관찰할 수 있는 것**:
- 루트 노드: 키 2개 (30, 60), 자식 3개
- 남부 노드: 키 2개 또는 3개
- 모든 리프 노드는 같은 깊이 (균형 트리!)

---

### 1.3 수학적 정의

#### 정의: 차수(order)가 $t$인 B-Tree

B-Tree $T$는 다음 **5가지 성질**을 만족하는 균형 탐색 트리이다:

| 성질 | 수학적 표현 | 의미 |
|------|-------------|------|
| **(1) 노드 키 개수 범위** | $t-1 \leq |keys| \leq 2t-1$ | 각 노드는 최소 $t-1$개, 최대 $2t-1$개의 키를 가진다 |
| **(2) 루트 특수 규칙** | $1 \leq |keys|_{root} \leq 2t-1$ | 루트는 최소 1개의 키를 가진다 |
| **(3) 자식 수 관계** $|children| = |keys| + 1$ | $k$개의 키를 가진 노드는 $k+1$개의 자식을 가진다 |
| **(4) 키 순서** | $k_1 < k_2 < ... < k_m$ | 노드 내 키는 오름차순으로 정렬 |
| **(5) 균형 성질** | $\forall$ leaf: $depth(leaf) = h$ | 모든 리프 노드는 같은 깊이 $h$에 있다 |

#### 키 범위의 직관적 이해

```
차수 t = 3인 경우:
┌─────────────────────────────────────────┐
│  최소 키: t-1 = 2개                    │
│  최대 키: 2t-1 = 5개                   │
│  최소 자식: t = 3개                    │
│  최대 자식: 2t = 6개                   │
└─────────────────────────────────────────┘
```

**왜 $t-1$과 $2t-1$인가?**
- **최소 $t-1$개**: 노드가 너무 비어있지 않도록 보장 (공간 효율성)
- **최대 $2t-1$개**: 노드가 꽉 차면 분할(split) 발생 → 트리 성장

---

### 1.4 수학적 표현: 노드의 구조

노드 $N$은 다음과 같이 표현된다:

$$N = (n, [c_0, k_1, c_1, k_2, c_2, ..., k_n, c_n])$$

여기서:
- $n$: 노드에 저장된 키의 개수 ($t-1 \leq n \leq 2t-1$)
- $k_i$: $i$번째 키 ($k_1 < k_2 < ... < k_n$)
- $c_i$: $i$번째 자식 포인터

**검색 트리 성질**:
$$\forall x \in subtree(c_{i-1}): x < k_i < \forall y \in subtree(c_i)$$

즉, $k_i$의 왼쪽 서브트리의 모든 키는 $k_i$보다 작고, 오른쪽 서브트리의 모든 키는 $k_i$보다 크다.

## 2. B-Tree의 높이 분석

### 2.1 직관적 이해: 왜 높이가 로그 시간인가?

**핵심 통찰**: B-Tree는 각 레벨에서 "최소 $t$배"씩 분기한다!

```
레벨 0 (루트):     최소 1개 노드
레벨 1:            최소 t개 노드  
레벨 2:            최소 t²개 노드
레벨 h:            최소 t^h개 노드 (리프)
```

> 각 레벨마다 최소 $t$배씩 늘어나므로, $n$개의 키를 저장하려면 높이가 로그 시간이 된다.

---

### 2.2 구체적 예시: t=2, n=7인 경우

```
최소 키 개수로 채운 B-Tree (t=2):

        [20]                    ← 레벨 0: 1개 노드, 1개 키
       /    \
    [10]    [30, 40]            ← 레벨 1: 2개 노드, 3개 키
    /  \    /   |   \
  [5] [15] [25] [35] [45,50]    ← 레벨 2: 5개 노드, 5개 키

총 키 개수: 1 + 3 + 5 = 9개 (실제로는 7개만 저장)
높이: h = 2
```

**관찰**: 높이가 2일 때 최대 $2 \cdot 2^3 - 1 = 15$개 키를 저장할 수 있다.

---

### 2.3 수학적 유도: 높이의 상한

#### 정리: B-Tree의 높이 상한

$n$개의 키를 저장하는 차수 $t$의 B-Tree의 높이 $h$는 다음을 만족한다:

$$h \leq \log_t \frac{n+1}{2}$$

#### 증명

**Step 1: 각 레벨의 최소 노드 수**

최소 키 개수로 채운 B-Tree를 고려하자:

| 레벨 | 최소 노드 수 | 해당 레벨의 최소 키 수 |
|------|--------------|------------------------|
| 0 (루트) | 1 | 1 |
| 1 | 2 | $2(t-1)$ |
| 2 | $2t$ | $2t(t-1)$ |
| 3 | $2t^2$ | $2t^2(t-1)$ |
| $i$ | $2t^{i-1}$ | $2t^{i-1}(t-1)$ |

**Step 2: 전체 키 개수의 하한**

높이가 $h$인 B-Tree의 최소 키 개수:

$$n \geq 1 + \sum_{i=1}^{h} 2t^{i-1}(t-1)$$

**Step 3: 기하급수 수열의 합**

$$n \geq 1 + 2(t-1) \sum_{i=0}^{h-1} t^i$$

$$n \geq 1 + 2(t-1) \cdot \frac{t^h - 1}{t - 1}$$

$$n \geq 1 + 2(t^h - 1)$$

$$n \geq 2t^h - 1$$

**Step 4: 높이 $h$에 대해 정리**

$$n + 1 \geq 2t^h$$

$$\frac{n+1}{2} \geq t^h$$

$$\log_t \frac{n+1}{2} \geq h$$

따라서:

$$\boxed{h \leq \log_t \frac{n+1}{2}}$$

---

### 2.4 직관과 수학의 연결

```
┌─────────────────────────────────────────────────────────────┐
│  높이 상한의 의미                                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  h ≤ log_t((n+1)/2) = (log_2((n+1)/2)) / (log_2(t))        │
│                                                             │
│  → t가 커질수록 (한 노드에 더 많은 키를 저장)               │
│    높이가 작아져서 검색이 빨라진다!                         │
│                                                             │
│  예시: n = 1,000,000 (100만 개 키)                         │
│  • t = 2 (이진 트리와 유사): h ≤ log_2(500000) ≈ 19        │
│  • t = 100: h ≤ log_100(500000) ≈ 3                        │
│                                                             │
│  → 디스크 접근 횟수가 19번 → 3번으로 줄어든다!             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 연산의 시간 복잡도 분석

### 3.1 검색 (Search) 연산

#### 직관적 이해

검색은 루트에서 시작하여 리프까지 낙하하며 진행된다:
1. 현재 노드에서 키를 선형 또는 이진 검색
2. 찾는 키가 없으면 적절한 자식으로 이동
3. 리프에 도달할 때까지 반복

#### 수학적 분석

**알고리즘**: B-TREE-SEARCH($x$, $k$)
- $x$: 현재 노드
- $k$: 찾으려는 키

```
B-TREE-SEARCH(x, k):
    i = 1
    while i ≤ x.n and k > x.key[i]:
        i = i + 1
    if i ≤ x.n and k == x.key[i]:
        return (x, i)           // 키 발견
    if x is leaf:
        return NIL              // 키 없음
    DISK-READ(x.c[i])           // 자식 노드 읽기
    return B-TREE-SEARCH(x.c[i], k)
```

**시간 복잡도**:

$$T_{search}(n) = O(h) = O(\log_t n) = O(\log n)$$

**세부 분석**:
- 각 노드 내 검색: $O(t)$ (선형) 또는 $O(\log t)$ (이진)
- 노드 방문 수: $O(h) = O(\log_t n)$
- 총 시간: $O(t \cdot \log_t n) = O(t \cdot \frac{\log n}{\log t})$

$t$가 상수라고 가정하면: $\boxed{O(\log n)}$

---

### 3.2 삽입 (Insert) 연산

#### 직관적 이해

**핵심 아이디어**: "선택적 분할" (proactive splitting)

```
삽입 과정:
1. 적절한 리프 노드 찾기 (검색과 동일)
2. 리프에 여유 공간 있음 → 바로 삽입 ✓
3. 리프가 가득 참 (2t-1개 키) → 분할 필요!
   
   분할 과정:
   ┌─────────────────────────────────────┐
   │  [10, 20, 30, 40, 50] (t=3, 꽉 참) │
   │              ↓ 분할                │
   │   [20]          [10]    [30,40,50] │
   │  /    \    →    /  \    /   |   \  │
   │              (새 부모)  (두 자식)  │
   └─────────────────────────────────────┘
   
   중간 키(20)가 부모로 올라가고, 나머지가 두 노드로 분할
```

#### 수학적 분석

**알고리즘**: B-TREE-INSERT($T$, $k$)

```
B-TREE-INSERT(T, k):
    r = T.root
    if r.n == 2t-1:              // 루트가 가득 찬 경우
        s = ALLOCATE-NODE()
        T.root = s
        s.leaf = FALSE
        s.n = 0
        s.c[1] = r
        B-TREE-SPLIT-CHILD(s, 1)  // 루트 분할
    B-TREE-INSERT-NONFULL(T.root, k)
```

**시간 복잡도**:

$$T_{insert}(n) = O(h) = O(\log_t n) = O(\log n)$$

**세부 분석**:
- 삽입 위치 탐색: $O(h)$
- 분할 연산: 한 레벨에서 최대 1회, $O(t)$ 시간
- 총 분할 횟수: 최대 $h$회
- 총 시간: $O(t \cdot h) = O(t \cdot \log_t n)$

$t$가 상수라고 가정하면: $\boxed{O(\log n)}$

---

### 3.3 삭제 (Delete) 연산

#### 직관적 이해

삭제는 가장 복잡한 연산으로, 세 가지 경우를 처리해야 한다:

```
┌─────────────────────────────────────────────────────────────┐
│  경우 1: 삭제할 키가 리프 노드에 있음                       │
│  → 바로 삭제 (단, 노드가 최소 키 개수보다 많아야 함)        │
├─────────────────────────────────────────────────────────────┤
│  경우 2: 삭제할 키가 남부 노드에 있음                       │
│  → 전임자(predecessor)나 후임자(successor)로 대체          │
├─────────────────────────────────────────────────────────────┤
│  경우 3: 삭제 후 노드가 underflow (키 < t-1)                │
│  → 형제 노드에서 재분배(borrow) 또는 병합(merge)           │
└─────────────────────────────────────────────────────────────┘
```

**재분배 vs 병합 예시** (t=3):

```
재분배 (Borrowing):
        [30, 60]                          [25, 60]
       /   |   \                         /   |   \
  [10,20] [40,50] [70,80]   →     [10,20] [30,40] [70,80]
  
  왼쪽 형제 [10,20]에서 20을 빌려오고, 부모 30이 남아있는 자리로

병합 (Merging):
        [30, 60]                          [60]
       /   |   \                         /    \
  [10]   [40,50]  [70,80]   →    [10,30,40,50]  [70,80]
  
  왼쪽 형제 [10]와 [40,50]를 부모 30과 합쳐서 하나의 노드로
```

#### 수학적 분석

**시간 복잡도**:

$$T_{delete}(n) = O(h) = O(\log_t n) = O(\log n)$$

**세부 분석**:
- 삭제할 키 탐색: $O(h)$
- 재분배 연산: $O(t)$ (한 레벨에서 키 이동)
- 병합 연산: $O(t)$ (한 레벨에서 노드 합치기)
- 최악의 경우: 루트까지 병합이 전파될 수 있음 → $O(t \cdot h)$

$t$가 상수라고 가정하면: $\boxed{O(\log n)}$

## 4. 시간 복잡도 요약

### 4.1 빅오 표기법 정리

| 연산 | 시간 복잡도 | 설명 |
|------|-------------|------|
| **검색** | $O(\log n)$ | 루트에서 리프까지 한 경로만 탐색 |
| **삽입** | $O(\log n)$ | 탐색 + 최대 $h$회 분할 |
| **삭제** | $O(\log n)$ | 탐색 + 재분배/병합 |
| **최소값** | $O(\log n)$ | 왼쪽 자식을 따라 리프까지 |
| **최대값** | $O(\log n)$ | 오른쪽 자식을 따라 리프까지 |
| **순회** | $O(n)$ | 중위 순회로 모든 키 방문 |

### 4.2 공간 복잡도

$$S(n) = O(n)$$

- $n$개의 키를 저장하는 데 $O(n)$ 개의 노드 사용
- 각 노드는 $O(t)$ 공간 사용
- 포인터 오버헤드: $O(n)$ (자식 포인터)

## 5. B-Tree vs 이진 탐색 트리 비교

```
┌────────────────────────────────────────────────────────────────┐
│                    B-Tree  vs  BST (이진 탐색 트리)            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  특성              B-Tree                 BST                  │
│  ─────────────────────────────────────────────────────────────│
│  분기 계수         t ~ 2t                 2                    │
│  높이             O(log_t n)             O(log n) ~ O(n)       │
│  균형 보장         항상 균형              AVL/RB-Tree 필요     │
│  디스크 효율성     높음 (한 노드 = 한 블록)  낮음              │
│  구현 복잡도       복잡                   단순                 │
│  주요 용도         데이터베이스, 파일시스템  메모리 내 탐색    │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 6. 핵심 정리

### B-Tree의 수학적 정의 (한눈에 보기)

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  차수 t의 B-Tree T는 다음을 만족:                              │
│                                                                 │
│  1. 모든 노드 N: t-1 ≤ |keys(N)| ≤ 2t-1                        │
│     (루트 제외: 1 ≤ |keys(root)| ≤ 2t-1)                       │
│                                                                 │
│  2. |children(N)| = |keys(N)| + 1                              │
│                                                                 │
│  3. 모든 리프 노드: depth(leaf) = h (균형)                     │
│                                                                 │
│  4. 높이 상한: h ≤ log_t((n+1)/2)                              │
│                                                                 │
│  5. 연산 시간: Search, Insert, Delete = O(log n)               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 핵심 통찰

> **B-Tree의 핵심은 "팬아웃(fan-out)"이다.**
> 
> 차수 $t$를 크게 잡으면 한 노드에 많은 키를 저장할 수 있고,
> 이는 트리의 높이를 줄여 디스크 접근 횟수를 최소화한다.
> 
> 수학적으로: $h = O(\log_t n)$ → $t$가 커질수록 $h$는 작아진다!

---

*이 문서는 B-Tree의 수학적 기초를 다룹니다. 구현 세부사항과 고급 변형(B+ Tree, B* Tree 등)은 별도 문서에서 다룹니다.*
