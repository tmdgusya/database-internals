# B-Tree 삭제와 Merge 완벽 가이드

> m=3 (3차 B-Tree) 기준으로 설명합니다.
> - 노드당 최대 키: 2개 (m-1)
> - 노드당 최소 키: 1개 (⌈m/2⌉ - 1)
> - 키가 0개가 되면 **언더플로우**

---

## 1. 삭제의 전체 흐름

삭제는 항상 이 순서를 따릅니다:

```
1. 삭제할 키를 찾는다
2. 리프에서 키를 제거한다
   (내부 노드의 키라면 → 후임자로 교체 후 리프에서 제거)
3. 언더플로우가 발생했는가?
   ├── 아니오 → 끝
   └── 예 → 재조정
         ├── 형제에게 빌릴 수 있는가?
         │    └── 예 → 빌려오기 (Borrow/Rotate)
         └── 아니오 → 병합 (Merge)
              └── 부모도 언더플로우? → 재귀적으로 반복
```

이 문서에서는 **병합(Merge)** 에 집중합니다.

---

## 2. 병합이 일어나는 조건

두 가지가 **동시에** 만족될 때:

1. 키 삭제 후 노드가 언더플로우 (키 0개)
2. 왼쪽/오른쪽 형제 모두 키가 최소 (1개) → 빌려줄 여유 없음

---

## 3. 병합의 핵심 아이디어

**병합 = 형제 + 부모의 구분 키 + 나 → 하나의 노드로 합치기**

이게 전부입니다. 하나씩 풀어보겠습니다.

---

## 4. 단순한 예시로 시작

### 초기 상태

```
         [20]
        /    \
     [10]    [30]
```

- 부모: keys=[20], children=[왼쪽, 오른쪽]
- 왼쪽 자식: keys=[10] (최소)
- 오른쪽 자식: keys=[30] (최소)

### 10을 삭제하면?

**Step 1**: 리프에서 10을 제거

```
         [20]
        /    \
      []     [30]       ← 왼쪽이 빈 노드! 언더플로우!
```

**Step 2**: 빌릴 수 있나? → 오른쪽 형제 [30]도 키가 1개 = 최소 → 못 빌림

**Step 3**: 병합 실행

병합은 3개를 하나로 합칩니다:

```
합칠 재료:
  - 언더플로우 노드의 남은 키: [] (없음)
  - 부모의 구분 키: 20
  - 형제의 키: [30]

합친 결과: [20, 30]
```

```
병합 후:
      [20 | 30]     ← 이 노드가 새 루트
```

부모가 구분 키(20)를 내려줬으므로 부모는 키가 0개가 됩니다.
부모가 루트였고 키가 0개 → **루트를 합쳐진 자식으로 교체** → 트리 높이 감소!

---

## 5. 병합 과정을 배열 수준에서 추적

위 예시를 코드 관점에서 봅시다.

### 병합 전 메모리 상태

```
parent.keys     = [20]
parent.children = [left, right]

left.keys       = []        ← 언더플로우!
left.children   = []

right.keys      = [30]
right.children  = []
```

### 병합 연산 (왼쪽과 오른쪽을 합침)

```python
# 1. 부모에서 구분 키를 가져온다
separator = parent.keys[0]           # 20

# 2. 왼쪽 노드에 구분 키를 추가
left.keys.append(separator)          # left.keys = [20]

# 3. 오른쪽 형제의 키를 왼쪽에 전부 붙인다
left.keys.extend(right.keys)         # left.keys = [20, 30]

# 4. 오른쪽 형제의 자식도 붙인다 (리프면 없음)
left.children.extend(right.children) # left.children = []

# 5. 부모에서 구분 키와 오른쪽 자식 포인터를 제거
parent.keys.pop(0)                   # parent.keys = []
parent.children.pop(1)               # parent.children = [left]
```

### 병합 후 메모리 상태

```
parent.keys     = []           ← 부모도 언더플로우!
parent.children = [left]

left.keys       = [20, 30]    ← 합쳐진 노드
left.children   = []
```

parent가 루트이므로 → `root = left`로 교체 → 높이 1 감소

## 6. 좀 더 복잡한 예시 — 3단 트리

### 초기 상태 (m=3)

```
              [20]
            /      \
        [10]        [30 | 40]
       /    \      /    |    \
     [5]   [15]  [25]  [35]  [45]
```

배열로 보면:

```
root.keys          = [20]
root.children      = [A, B]

A.keys             = [10]
A.children         = [a1, a2]

B.keys             = [30, 40]
B.children         = [b1, b2, b3]

a1.keys = [5]    a2.keys = [15]
b1.keys = [25]   b2.keys = [35]   b3.keys = [45]
```

### 5를 삭제

**Step 1**: 리프 a1에서 5 제거 → a1.keys = [] → 언더플로우!

```
              [20]
            /      \
        [10]        [30 | 40]
       /    \      /    |    \
    [ ]    [15]  [25]  [35]  [45]
     ↑ 언더플로우!
```

**Step 2**: 형제 a2=[15]는 키 1개 = 최소 → 빌릴 수 없음 → **병합!**

```
합칠 재료:
  - a1의 키: [] (없음)
  - 부모 A의 구분 키: 10
  - 형제 a2의 키: [15]

합친 결과: [10, 15]
```

a1과 a2를 합치고, 부모 A에서 구분 키 10을 제거:

```
A.keys     = []          ← A도 언더플로우!
A.children = [merged]    (merged.keys = [10, 15])
```

**Step 3**: A가 언더플로우 → 형제 B=[30,40]은 키 2개, 여유 있음! → **빌려오기**

```
빌려오기 전:
              [20]
            /      \
          []        [30 | 40]
         /         /    |    \
   [10|15]      [25]  [35]  [45]

빌려오기:
  1. 부모(root)의 구분 키 20을 A로 내림
  2. 형제 B의 첫 번째 키 30을 부모로 올림
  3. B의 첫 번째 자식 [25]를 A로 이동
```

```
빌려오기 후:
              [30]
            /      \
        [20]        [40]
       /    \      /    \
  [10|15]  [25]  [35]  [45]
```

A.keys=[20], A.children=[merged, b1] → 정상!

## 7. 병합이 연쇄적으로 일어나는 경우

### 초기 상태 (m=3)

```
              [20]
            /      \
        [10]        [30]
       /    \      /    \
     [5]   [15]  [25]  [35]
```

### 5를 삭제

**Step 1**: a1=[5] → 키 제거 → [] → 언더플로우

**Step 2**: 형제 a2=[15] 최소 → 병합

```
합침: [] + 10(구분키) + [15] → [10, 15]

A.keys = []  ← 언더플로우!
```

**Step 3**: A 언더플로우 → 형제 B=[30]도 최소 → 또 병합!

```
합침: A의 남은 것 + 20(구분키) + B의 키[30]

부모(root) 수준에서:
  합칠 재료:
    - A의 키: [] (없음)
    - root의 구분 키: 20
    - B의 키: [30]

  합친 결과: [20, 30]  ← 새 내부 노드

  자식도 합침:
    A의 자식 [merged] + B의 자식 [b1, b2]
    = [[10,15], [25], [35]]
```

```
결과:
           [20 | 30]          ← 새 루트 (높이 감소!)
          /    |    \
    [10|15]  [25]  [35]
```

root가 키 0개 → 합쳐진 노드가 새 루트

## 8. 병합의 규칙 정리

### 입력

```
parent: 부모 노드
i:      언더플로우가 발생한 자식의 인덱스
```

### 알고리즘 (오른쪽 형제와 병합하는 경우)

```
underflow_child = parent.children[i]
right_sibling   = parent.children[i + 1]
separator       = parent.keys[i]

1. underflow_child에 separator 추가
2. underflow_child에 right_sibling의 모든 키 추가
3. underflow_child에 right_sibling의 모든 자식 추가
4. parent.keys에서 separator 제거
5. parent.children에서 right_sibling 제거
6. parent.key_count 감소
```

### 알고리즘 (왼쪽 형제와 병합하는 경우)

```
left_sibling    = parent.children[i - 1]
underflow_child = parent.children[i]
separator       = parent.keys[i - 1]

1. left_sibling에 separator 추가
2. left_sibling에 underflow_child의 모든 키 추가
3. left_sibling에 underflow_child의 모든 자식 추가
4. parent.keys에서 separator 제거
5. parent.children에서 underflow_child 제거
6. parent.key_count 감소
```

핵심 차이: **항상 왼쪽 노드에 합친다** (왼쪽이 살아남음)

### 병합 후 확인

```
parent.key_count가 0이면:
  ├── parent가 루트 → 합쳐진 자식이 새 루트 (높이 감소)
  └── parent가 루트 아님 → parent도 언더플로우 → 재귀적으로 처리
```

---

## 9. 구분 키(Separator)가 왜 필요한가?

이 부분이 병합에서 가장 헷갈리는 포인트입니다.

```
        [20]
       /    \
    [10]    [30]
```

왼쪽 자식의 키는 모두 < 20, 오른쪽 자식의 키는 모두 > 20

만약 구분 키 없이 그냥 합치면:

```
[10, 30]  ← 10과 30 사이의 값(예: 15)은 어디로?
```

구분 키 20을 사이에 넣어야 정렬이 유지됩니다:

```
[10, 20, 30]  ← 올바른 순서
```

부모의 구분 키는 원래 **두 자식의 경계 값** 역할을 했으므로,
합칠 때 반드시 **사이에** 넣어야 B-Tree 속성이 유지됩니다.

---

## 10. 빌려오기 vs 병합 — 판단 기준

```
언더플로우 발생!

왼쪽 형제가 있고 키 >= 2개? → 왼쪽에서 빌려오기
오른쪽 형제가 있고 키 >= 2개? → 오른쪽에서 빌려오기
둘 다 안 됨? → 병합
  ├── 왼쪽 형제가 있으면 → 왼쪽과 병합
  └── 없으면 → 오른쪽과 병합
```

빌려오기를 먼저 시도하는 이유: 빌려오기는 부모의 키 개수가 변하지 않아서
**연쇄 반응이 없습니다**. 병합은 부모의 키가 줄어서 연쇄될 수 있습니다.

---

## 11. 요약 — 병합을 한 문장으로

> **형제와 나 사이에 있던 부모의 구분 키를 내려서,
> 형제 + 구분 키 + 나를 하나의 노드로 합친다.
> 부모는 키 하나와 자식 하나를 잃는다.**
